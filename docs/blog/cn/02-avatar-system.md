# Avatar 系统：一个有灵魂的桌面宠物

> 它不只是一个图标。它会走、会跑、会打哈欠、会生气——它是你在桌面上能"摸"得到的 AI。

---

## 不只是装饰

市面上的 AI 助手通常以两种形态存在：聊天窗口或侧边栏。它们在你需要时出现，不需要时消失。Bonio 的 Avatar 是第三种形态：**它一直都在**。

它浮动在你当前窗口的顶边中央，像一个坐在窗台上晃着腿的小家伙。你切换窗口，它就跟过去；你把窗口最小化，它就落到任务栏上；你全屏工作，它安静地缩在角落。它的存在本身就是一个信号：**AI 在陪着你，不需要你去找它**。

这种"窗口边缘锚定"设计经过了深思熟虑——它不占据窗口内部空间，不与应用的标题栏、标签页、工具栏冲突。它"坐在"窗口的**外部上沿**，就像一只猫趴在显示器顶上。

---

## 双引擎架构：为什么 Avatar 是独立窗口

Avatar 跑在一个**独立的 Flutter 引擎**中，通过 `desktop_multi_window` 创建为操作系统级的 **透明浮窗**（Windows 上 `WS_POPUP`，macOS 上无边框透明 NSWindow）。它与 Bonio 主窗口完全解耦：

```
Main Engine (Bonio 窗口)              Avatar Engine (浮动窗口)
┌──────────────────────┐              ┌──────────────────────┐
│  AppState            │   sync()     │  AvatarFloatingApp    │
│  ├─ NodeRuntime      │─────────────▸│  ├─ Lottie 动画       │
│  │  ├─ ChatController│  (JSON)      │  ├─ 交互状态机        │
│  │  ├─ AvatarCtrl    │              │  ├─ 窗口锚定/弹性跟随  │
│  │  └─ STT Manager   │              │  └─ 气泡/文本输入      │
│  └─ TrayService      │              └──────────────────────┘
└──────────────────────┘
```

这样设计的好处：

1. **主窗口最小化时 Avatar 仍然可见。** 它是独立的操作系统窗口，不受主窗口生命周期影响。
2. **Avatar 可以覆盖到其他应用上方。** `alwaysOnTop + skipTaskbar`，不抢占焦点。
3. **渲染完全隔离。** Avatar 的动画、交互不会阻塞主窗口的聊天、配置等操作。
4. **跨引擎状态同步。** 主引擎序列化 `AvatarSnapshot` 为 JSON，通过 `invokeMethod('sync', json)` 推送到 Avatar 引擎。所有状态决策在主引擎完成，Avatar 引擎只负责渲染。

## 窗口锚定：坐在窗口边缘的物理学

Avatar 的位置由**窗口锚定状态机**管理：

| 状态 | 行为 | 触发条件 |
|------|------|---------|
| `ANCHORED_WINDOW` | 坐在活动窗口顶边中央 | 默认；有普通窗口在前台 |
| `FULLSCREEN_CORNER` | 缩到屏幕右上角 | 活动窗口全屏（覆盖率 > 95%） |
| `USER_OFFSET` | 维持用户拖拽后的偏移位置 | 用户手动拖拽 Avatar |
| `ON_DOCK` | 落在任务栏/Dock 上 | 所有窗口最小化或关闭 |

窗口切换采用**即时跟踪**——用户切换到哪个窗口，Avatar 立刻传送到新窗口顶边，不拖泥带水。但窗口的移动追踪则带有**弹簧物理感**：

```dart
// 60fps 渲染循环中的弹性跟随
_currentX += (_targetX - _currentX) * 0.12;  // ~200ms 平滑到位
_currentY += (_targetY - _currentY) * 0.12;
```

Avatar 不会瞬间闪现到新位置，而是以 200ms 左右的弹性跟随效果滑过去。这种细节让 Avatar 的移动有了"物理重量"——它不是一个贴图，而是一个有惯性的小生物。

## 交互设计：5 种手势的分层状态机

Avatar 的浮动窗口上，一块透明区域承载了 5 种交互。因为 `windowManager.startDragging()` 会抢占 OS 级指针，Flutter 自带的 `GestureDetector` 无法正确区分这些手势。所以 Avatar 使用**原始 `Listener` + 指针状态机**自行判别：

```
PointerDown (左键)
  ├─ 移动 > 2px 且在 300ms 内 → DRAG（窗口拖拽）
  ├─ 300ms 计时器到期（未移动）→ VOICE START（开始录音）
  │     └─ PointerUp → VOICE STOP + 语音识别完成
  └─ PointerUp < 300ms → CLICK_PENDING
       ├─ 400ms 内第二次 PointerDown → DOUBLE_CLICK（文本输入框）
       └─ 400ms 超时 → SINGLE_CLICK（随机动画+气泡）

PointerDown (右键, buttons & 0x02)
  └─ CONTEXT MENU（弹出右键菜单）
```

5 种交互各有不同的设计意图：

| 交互 | 行为 | 设计逻辑 |
|------|------|---------|
| **单击** | 随机动画 + 情绪气泡（颜文字） | 低频交互，消解压力。猫咪打哈欠、伸懒腰、甩尾巴 |
| **双击** | 弹出迷你文本输入框 | 高频执行动作。桌面用户习惯双击打开，唤起输入 |
| **长按** | 按住说话（Push to Talk） | 映射"对讲机"隐喻。按住说，松开发送，减少误录风险 |
| **拖拽** | 手动位移 | 让用户能挪开它。挪动后在该窗口锁定相对位置 |
| **右键** | 功能工具箱 | 用户习惯右键寻找"怎么办"，核心 AI 能力入口 |

## 动画状态系统

Avatar 的动画分为三个层级，按**优先级**叠加：

### 1. 手势动作（最高优先级）
短时播放，覆盖一切。用于服务器指令触发的操作动画：

| 动作 | 动画 | 场景 |
|------|------|------|
| `tapping` | 敲击屏幕 | 点击/轻触操作 |
| `swiping` | 滑动 | 滚动/翻页 |
| `typing` | 打字 | 键盘输入 |
| `launching` | 启动 | 打开应用 |
| `takingphoto` | 拍照 | 相机操作 |

### 2. 运动状态
Avatar 的物理移动方式：

| 状态 | 动画 | 触发 |
|------|------|------|
| `walking` | 走路 | 窗口顶边随机漫步（10-30s 间隔，2-4 段折线） |
| `running` | 奔跑 | 服务端 `moveTo(mode: "run")` |
| `dragging` | 被拖拽 | 用户拖拽中 |

### 3. 活动状态（基础层）
Avatar 的"情绪"和"智能状态"：

| 状态 | 动画 | 触发 |
|------|------|------|
| `idle` | 待机呼吸 | 默认 |
| `listening` | 竖起耳朵 | 用户长按录音中 |
| `thinking` | 沉思 | LLM 请求进行中（无 tool call） |
| `working` | 忙碌 | Tool call 执行中 |
| `speaking` | 说话 | TTS 朗读回复内容 |
| `happy` | 开心 | 单击随机动作 / 操作成功 |
| `bored` | 无聊 | 单击随机动作 |
| `confused` | 困惑 | 单击随机动作 |
| `angry` | 生气 | 不支持的格式 / 操作失败 |
| `sleeping` | 睡觉 | 服务端指令 |

动画渲染使用 **Lottie**（`lottie` 包），每个状态对应一个 JSON 动画文件。主题系统通过 `theme.json` 定义状态到动画文件的映射，换主题 = 换一整套 Lottie 文件 + 配置文件。

## 空闲漫步：让它"活"起来

Avatar 在用户长时间不互动时会**自己散步**——这是让它有"灵魂感"的关键设计。

漫步只在**同时满足全部条件**时触发：
1. 没有交互进行中（菜单关闭、输入框隐藏、Lens 未激活）
2. 锚定在窗口顶边或任务栏（`ANCHORED_WINDOW` 或 `ON_DOCK`）
3. 距离上次漫步结束 10-30 秒

每次漫步包含 **2-4 段短途折线**，每段之间停顿 0.5-1.5 秒。一旦用户有任何交互——单击、双击、长按、拖拽、焦点窗口切换——漫步**立即取消**，零延迟。这保证了 Avatar 永远是"可打断的"，不会在用户想用它的时候还在自顾自地走。

## 气泡系统

Avatar 头顶的对话气泡是主要的信息反馈通道：

- **3 行上限**，超出自动滚动（提词器效果）
- 内容来源多样：语音识别实时字幕、AI 回复流式文本、TTS 朗读内容、单击情绪颜文字、服务端 `setBubble` 指令
- 支持**自定义背景色和文字色**（服务端可以通过颜色传达不同的信息层级）
- 支持**倒计时模式**（来电振铃倒计时等场景）

## 服务端驱动：AI 也能操控它

Avatar 不只是用户的宠物——**AI 也能控制它**。通过 Gateway 协议的 `avatar.command` 事件，AI 可以向 Avatar 发送指令：

| 指令 | 效果 |
|------|------|
| `setState("happy")` | 切换动画状态 |
| `moveTo(x, y, "walk")` | 走到指定位置 |
| `setBubble("找到了！")` | 显示气泡 |
| `tts("好的，已帮你完成")` | 朗读文字 |
| `playSound("notification")` | 播放提示音 |
| `performAction("tap", x, y)` | 做敲击动作 |
| `sequence([...])` | 串行执行多个指令 |

这意味着 AI 不仅能通过文字回复你，还能通过 Avatar 的**动作、表情、声音**来传达信息。这是 Bonio 与传统聊天机器人最本质的区别——**AI 有了"身体语言"**。

## 主题系统：不只是换皮

Avatar 的外观由 **主题目录** 定义：

```
assets/themes/default-cat/
├── theme.json          # 状态→动画映射 + bottomInsets 等参数
└── cat-*.json          # 34 个 Lottie 动画文件
```

`theme.json` 定义了每个 Activity/Motion/Gesture 状态对应的 Lottie 文件路径，以及动画的底部插入值（`bottomInsets`，用于调整动画在窗口中的位置）。替换主题就是替换整个目录——社区可以创作新角色（狗、机器人、像素小人），只需提供 Lottie 动画文件 + 配置即可。
