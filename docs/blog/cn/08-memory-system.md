# 记忆系统：你的外挂大脑

> 聊天记录会丢失，但 Bonio 的记忆不会。你喂给它的每一张截图、每一段文字、每一个文件，都会被自动分类、打标签、写摘要，然后在需要时精确召回。

---

## 记忆 vs 聊天历史

传统的 AI 聊天应用都有"历史记录"，但那只是**按时间排列的消息流**。你要找"上周提到的那家餐厅"，得在几十条聊天记录里翻。

Bonio 的记忆系统不同——它是**结构化的知识库**：

| | 聊天历史 | Bonio 记忆 |
|---|---|---|
| **存储方式** | 按时间顺序 | 按主题 + 标签索引 |
| **检索方式** | 滚动翻找 | 搜索 + 标签过滤 + 对话查询 |
| **内容形式** | 纯文本消息 | 截图 + 文本 + 文件 + AI 摘要 |
| **生命周期** | 会话结束即尘封 | 跨会话持久化，随用随取 |
| **AI 感知** | 每次手动引用 | 自动注入相关记忆到上下文 |

## 两种"输入"：截屏记与投喂记

### 右键截屏记

用户在任意窗口右键 Avatar，选择"记一记"：

1. Bonio 拍摄当前窗口的完整截图（`PrintWindow` API）
2. 保存截图 + 窗口标题 + 时间戳到本地 `boji-notes/` 目录
3. 后台调用 **独立的 LLM 会话**（`boji-notes` session）进行多模态分析
4. AI 返回结构化结果：`{"tags": ["#技术文档", "#架构"], "summary": "这是一篇关于微服务架构的设计文档..."}`
5. Avatar 气泡反馈："已存入 [#技术文档] 喵！"

这种交互的设计意图是**极低门槛**——用户看到任何想"记住"的东西，右键一点就行了。不需要打开笔记应用、不需要打字、不需要手动分类。AI 帮你完成所有的组织和索引工作。

### 拖拽投喂记

比右键更"自然"的交互：**直接把东西拖到 Avatar 身上。**

用户可以从任何地方拖拽内容到 Avatar：

| 拖拽来源 | 支持格式 | 实现 |
|---------|---------|------|
| 文件管理器 | 文件路径列表 (`CF_HDROP`) | Win32 COM `IDropTarget` |
| 文本选中 | Unicode 文本 (`CF_UNICODETEXT`) | Win32 COM `IDropTarget` |
| 剪贴板/应用 | 位图 (`CF_DIB`) | Win32 COM `IDropTarget` |

拖拽过程中的**动画反馈**经过精心设计，模拟"喂食"的拟人化体验：

```
感应期 (DragEnter/DragOver)
  → Avatar 切换到 openmouth 动画（嘴巴张大，身体前倾）

吸入期 (Drop)
  → Avatar 播放 eating 动画（2秒，闭嘴咀嚼）

消化期 (保存完成)
  → Avatar 播放 satisfied 动画（1秒，满足表情，摸摸肚子）

拒食期 (格式不支持/文件过大)
  → Avatar 播放 refuse 动画（皱眉、推手）
```

这种交互设计的精妙之处在于：**用户不需要"学会"怎么用**。你在桌面操作系统上拖文件拖了二十年，这是最本能的操作。Bonio 只是在这个本能操作的下游接上了 AI 分析能力。

## 数据存储结构

记忆数据存储在本地文件系统中（非 SQLite，保持轻量和可移植）：

```
~/.bonio/boji-notes/
├── index.json            # 笔记索引：id、标签、时间、来源应用
├── attachments/          # 原始附件（截图、拖拽的文件）
├── thumbnails/           # 缩略图（200px 宽，用于卡片网格展示）
└── note_*.json           # 各笔记的详细数据（摘要、标签、原始文本）
```

`index.json` 中的每条记录结构：

```json
{
  "id": "note_20260430_143021",
  "title": "Bonio 架构设计文档",
  "tags": ["#技术文档", "#架构", "#AI"],
  "summary": "讨论了 Bonio 的双引擎架构和插件系统设计...",
  "sourceApp": "Google Chrome",
  "createdAt": "2026-04-30T14:30:21Z",
  "attachments": ["note_20260430_143021_screenshot.png"]
}
```

## 智能分类：AI 驱动的自动打标

传统的笔记应用要求用户手动打标签、建文件夹。Bonio 把这个工作交给了 AI。

每次保存笔记时，系统使用一个**独立的 LLM 会话**（`boji-notes` session，不与用户的聊天历史混淆）进行内容分析。这个专用会话有自己的 system prompt，告诉 LLM 它的角色是"内容分类器"：

```
你是 BoJi Notes 分类助手。分析用户给出的内容，返回 JSON 格式：
{
  "tags": ["标签1", "标签2", ...],   // 3-5个中文标签，如 #购物、#美食、#技术文档
  "summary": "一句话内容摘要"
}
```

标签系统的好处：
1. **跨会话检索**：用户在聊天中说"帮我找 #购物 的笔记"，AI 能直接检索对应标签
2. **UI 过滤**：Memory 管理界面支持按标签筛选，点一下标签就过滤
3. **聚类发现**：用户可以发现自己"不知不觉记了好多 #美食 的东西"

## Memory 管理界面

主窗口中有一个 **Memory 标签页**，提供完整的笔记管理：

- **搜索**：全文检索笔记标题、摘要、标签
- **标签过滤**：点击标签 Chip，筛选该类笔记
- **卡片网格**：缩略图 + 标题 + 摘要 + 标签的卡片布局
- **详情查看**：点击卡片打开详情，查看原始截图和完整摘要
- **删除**：长按或右键删除笔记

这个界面的设计意图是：**用户的记忆资产是可视、可管理的**。它不是隐藏在某个角落的数据库，而是你随时可以浏览的"知识相册"。

## 对话式调取：用自然语言找回记忆

记忆系统的终极能力是**对话式检索**——你不需要打开 Memory 界面翻找，直接跟 Bonio 说：

- "帮我找上周记的那篇关于 Kubernetes 的文章"
- "我上个月好像记了一个关于红色背包的，帮我找找"
- "把最近所有 #购物 的笔记列出来"
- "上次开会提到的产品定价是多少？"

Bonio 会在 `boji-notes` 会话的上下文中检索相关记忆，以卡片形式展示结果。点击卡片可以追溯原始窗口或文件路径。

这项能力的实现依赖 HiClaw 后端的 **Memory 子系统**（`memory/memory.hpp`）：
- `store(key, content, category)` — 存储记忆（category: `core`/`daily`/`conversation`）
- `recall(query, limit)` — 语义检索记忆
- `forget(key)` — 删除记忆

记忆分为三个类别：
- **core**（核心记忆）：始终注入 LLM 上下文，存储用户偏好和重要事实
- **daily**（日常记忆）：当天相关的临时信息
- **conversation**（对话记忆）：对话过程中自动提取的关键信息

每次 Agent 启动时，`core` 类别的记忆会自动注入 system prompt，让 AI 始终"记得"关于用户的重要事项。

## 为什么不是数据库

你可能注意到整个系统用的是 JSON 文件，而不是 SQLite 或 PostgreSQL。这是有意为之：

1. **零依赖**：JSON 文件不需要任何数据库引擎。操作系统自带的文件系统就是"数据库"。
2. **可迁移**：用户可以轻松备份 `boji-notes/` 目录、同步到其他机器、甚至手动编辑。
3. **透明性**：用户完全拥有自己的数据。它们以可读的 JSON 格式存在于用户自己选择的目录中。
4. **规模考量**：个人记忆的数据量（通常几千条）在文件系统上完全可控。当数据量增长到需要数据库时，迁移也不复杂——JSON 格式天然适合批量导入任何数据库。
